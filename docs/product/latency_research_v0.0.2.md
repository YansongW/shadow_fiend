# shadow_fiend v0.0.2 延迟调研报告

**日期**：2026-06-25  
**背景**：用户反馈 PRD 中「端到端 1.5 秒」目标过高，实时字幕应以 200–300ms 为目标。

---

## 1. 当前 v0.0.1 延迟拆解

当前 pipeline：`音频捕获 → VAD 切句 → ASR → 翻译 → UI`

| 环节 | 当前实现 | 估算耗时 | 备注 |
|------|----------|----------|------|
| 音频采集 | PyAudio 100ms chunk | ~100ms | 可缩短到 20–40ms |
| VAD 等待完整句子 | 能量阈值 VAD，min_silence=350ms | **300–800ms** | 最大瓶颈：必须等一句话说完才送 ASR |
| ASR | SenseVoice-Small 整句 generate | 100–500ms | 10s 音频官方约 70ms，但整句调用有 overhead |
| 翻译 | Argos `ja/ko → en → zh` pivot | **1000–3000ms** | 最大瓶颈之二 |
| UI 刷新 | 100ms QTimer | <100ms | 可忽略 |
| **总计** | | **2–5 秒** | 与用户体感一致 |

**结论**：延迟不是某个单点问题，而是「整句式 batch 架构」本身决定的。要降到 300ms 以内，必须改成 **流式/增量 pipeline**。

---

## 2. 为什么 200–300ms 是合理目标

- 人类对话中，>250ms 的延迟会开始被感知为「不自然」
- 行业实时字幕/同传系统普遍以 **200–400ms** 为目标
  - Deepgram streaming ASR：<300ms
  - Alibaba RTS 实时字幕：200–400ms
  - Whisper-Streaming 论文：3.3 秒（偏高，基于整句 Whisper）
  - 低延迟直播翻译方案：目标 ≤280ms

> 注意：200–300ms 指的是「第一个字/词出现在字幕上」的延迟，不是整句翻译完成的时间。整句会随说话持续滚动更新。

---

## 3. 行业方案参考

### 3.1 低延迟 pipeline 通用结构

```
音频捕获（20–40ms chunk）
  → 低延迟 VAD（Silero VAD，32ms 级）
  → 流式 ASR（每收到一个 chunk 就输出 partial transcript）
  → 增量翻译（按短语/短句翻译，而非等整句）
  → 字幕渲染（流式更新）
```

### 3.2 ASR 选项对比

| 方案 | 延迟 | 本地 | 质量 | 中日韩支持 | 备注 |
|------|------|------|------|------------|------|
| **SenseVoice-Small（当前）** | 整句 100–500ms | ✅ | 高 | ✅ 优秀 | 非流式，需 chunk 策略 |
| **Paraformer（FunASR 流式）** | **<100ms** | ✅ | 高 | ✅ 中文极强 | FunASR 官方 streaming 方案 |
| **Whisper-Streaming** | 1–3 秒 | ✅ | 高 | ⚠️ 一般 | 基于 Whisper，偏慢 |
| **faster-whisper** | 100–300ms | ✅ | 高 | ⚠️ 一般 | 可流式 chunk 处理 |
| **Deepgram streaming** | **<300ms** | ❌ | 高 | ⚠️ | 违背本地隐私原则 |

### 3.3 翻译选项对比

| 方案 | 延迟 | 本地 | 中日韩直接 | 备注 |
|------|------|------|------------|------|
| **Argos（当前）** | 1–3s | ✅ | ❌ 需 en pivot | 质量差、延迟高 |
| **Helsinki-NLP/opus-mt** | 50–200ms | ✅ | ✅ ja/ko-zh | 模型小，快 |
| **NLLB-200-distilled-600M** | 100–400ms | ✅ | ✅ | 一个模型覆盖多语言 |
| **本地小 LLM（Qwen2.5-3B）** | 200–800ms | ✅ | ✅ | 质量高但慢 |
| **商业 API** | <100ms | ❌ | ✅ | 不考虑 |

## 4. 可行技术路径

### 方案 A：最小改动路径（目标 500–800ms）

**改动**：
- VAD chunk 从 100ms 降到 20–40ms
- VAD 参数更激进：min_silence=150ms, min_speech=100ms
- Argos 替换为 opus-mt 直接翻译
- ASR 仍用 SenseVoice 整句，但句子切得更短

**预期延迟**：500–800ms  
**优点**：改动小，风险低  
**缺点**：仍非真正实时，口语停顿多时会频繁刷新

### 方案 B：流式 ASR + 增量翻译（目标 200–300ms）⭐ 推荐

**改动**：
- 将 ASR 从 SenseVoice 整句改为 **Paraformer 流式识别**（FunASR 原生支持）
- 或保留 SenseVoice，但改为 **短 chunk 连续识别 + LocalAgreement 确认**
- VAD 改为 **Silero VAD**，32ms 级响应
- 翻译改为 **opus-mt 片段级翻译**：每收到一个语义完整的短语就翻译
- UI 改为流式更新：partial 原文和译文同步出现

**预期延迟**：200–400ms  
**优点**：真正接近实时，观影体验好  
**缺点**：
  - 工程复杂度高
  - Paraformer 流式模型需要调研和测试
  - 增量翻译可能出现语义不连贯（需后处理）

### 方案 C：混合路径（目标 300–500ms）

**改动**：
- 保留 SenseVoice，但用更短的音频窗口（如 500ms–1s）连续调用
- 用 Silero VAD 快速切分
- opus-mt 直接翻译
- 对短片段翻译结果做平滑/去重

**预期延迟**：300–500ms  
**优点**：不需要换 ASR 模型，兼容现有代码  
**缺点**：SenseVoice 反复调用开销大，长句准确性下降

---

## 5. PM 推荐方案

**推荐方案 B（流式 ASR + 增量翻译）**，但分阶段实施：

### Phase 1：v0.0.2 核心目标（4–6 周）
- 切换到 **Paraformer 流式 ASR**（FunASR 官方 streaming）
- 使用 **Silero VAD** 替代能量阈值 VAD
- 音频 chunk 缩短到 **32–40ms**
- 翻译改为 **opus-mt 直接模型**
- UI 支持 partial 字幕流式更新

**目标延迟**：300–500ms

### Phase 2：v0.0.3 优化目标
- 增量翻译 + 上下文缓存
- 模型量化/ONNX 加速
- 目标延迟：**200–300ms**

---

## 6. 关键问题待确认

1. **是否接受为达到 200–300ms 而将 ASR 从 SenseVoice 切换到 Paraformer 流式？**
   - SenseVoice 在静态准确性上有优势，但流式不是它的强项。
2. **是否接受 v0.0.2 先做到 300–500ms，v0.0.3 再优化到 200–300ms？**
3. **UI 是否支持「partial 字幕」（边说边出字，可能中间修正）？**
   - 这是流式 ASR 的自然行为，与当前「等一句完整再显示」不同。
4. **是否继续坚持以本地、隐私优先为原则，不考虑商业 API？**
