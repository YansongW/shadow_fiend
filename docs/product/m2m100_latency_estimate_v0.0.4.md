## 2. 基于文献与理论的延迟估算

### 2.1 模型规模对比

| 模型 | 参数量 | 典型 FP32 大小 | 架构 |
|---|---|---|---|
| opus-mt（当前） | ~80M | ~300MB | MarianMT，6+6 层 |
| **m2m-100** | **418M** | **~1.6GB** | Transformer，12+12 层 |
| NLLB-200 distilled | 600M | ~2.4GB | Transformer，更深的编解码器 |
| m2m-100 1.2B | 1.2B | ~4.8GB | 更深更宽 |

### 2.2 延迟估算方法

基于以下事实进行估算：

1. **opus-mt 当前实测延迟**（Apple Silicon CPU，greedy decoding）：
   - ja→en：~31 ms
   - en→zh：~72 ms
   - pivot 路径 ja→zh：约 100–150 ms

2. **m2m-100 418M 参数量是 opus-mt 的约 5 倍**，层数也更多（12 层 vs 6 层）

3. **CTranslate2 官方 benchmark** 显示：在 CPU 上，更大模型推理时间与参数量大致成正比

4. **文献中 m2m-100/NLLB 的推理时间**（服务器 CPU/GPU）：
   - m2m-100 418M 单句推理：约 200–500 ms（CPU）
   - NLLB-600M 单句推理：约 300–800 ms（CPU）

### 2.3 Apple Silicon CPU 估算结果

| 语言对 | opus-mt pivot | m2m-100 418M 直接 | 加速/减速 |
|---|---|---|---|
| ja→zh | 100–150 ms | **200–400 ms** | 更慢 1.5–3 倍 |
| ko→zh | 100–150 ms | **200–400 ms** | 更慢 1.5–3 倍 |
| en→zh | 72 ms | **150–300 ms** | 更慢 2–4 倍 |

**说明**：

- 直接模型消除了 pivot 的双倍调用，但单次推理更慢
- 在 Apple Silicon CPU 上，418M 模型推理时间可能接近或超过 300 ms/句
- 若使用 CTranslate2 加速 m2m-100，可能提升 30–50%，但仍慢于 opus-mt

## 3. 内存占用估算

| 模型 | 加载后内存占用 |
|---|---|
| opus-mt ja-en + en-zh（两个模型） | ~1.2GB |
| m2m-100 418M（一个模型） | **~1.5–2.0GB** |
| NLLB-600M（一个模型） | **~2.0–2.5GB** |

shadow_fiend 当前同时加载：SenseVoice + Silero VAD + RNNoise + 翻译模型。若替换为 m2m-100 418M，总内存可能接近或超过 4GB，在 8GB Mac 上压力较大。

## 4. 质量预期

| 模型 | ja→zh BLEU（通用领域） | 影视字幕领域 | 无标点短句 |
|---|---|---|---|
| opus-mt pivot | 中等 | 一般，有幻觉 | 易重复 |
| m2m-100 418M | 25–35 | 较好 | 可微调改善 |
| NLLB-600M | 30–40 | 较好 | 可微调改善 |

m2m-100/NLLB 作为多语言预训练模型，在通用领域质量优于 opus-mt pivot。但在影视字幕这种口语化、无标点领域，仍需 fine-tuning 才能达到最佳效果。

---

## 5. 关键结论

1. **m2m-100 418M 在 Apple Silicon CPU 上会比当前 opus-mt pivot 更慢**
   - 估算延迟：200–400 ms/句
   - 当前 opus-mt pivot：100–150 ms/句

2. **m2m-100 418M 的质量预期优于 opus-mt pivot**，但差距需要 fine-tuning 才能充分体现

3. **内存压力显著增加**：单个模型约 1.5–2GB

4. **若目标是"低延迟"，m2m-100 418M 不是直接解决方案**
   - 它解决的是"pivot 路径"和"直接语言对"问题
   - 但付出了单次推理更慢的代价

## 6. 对 v0.0.4 的启示

### 选项 A：直接微调 m2m-100 418M

- **优点**：一个模型支持多向翻译，消除 pivot
- **缺点**：本地推理延迟 200–400 ms，可能影响实时体验
- **适用**：对延迟要求不极致、更看重质量的用户

### 选项 B：微调后再蒸馏/量化

- **思路**：先用 m2m-100 418M 作为教师，蒸馏一个 100M–200M 的学生模型
- **优点**：保留直接语言对能力，同时降低延迟
- **缺点**：工程复杂，需要额外训练投入
- **预期延迟**：80–150 ms/句（接近当前 opus-mt）

### 选项 C：继续优化 opus-mt + 延迟容忍设计

- **思路**：保持当前架构，通过工程优化（缓存、批处理）和 UX 设计（如翻译异步显示）来缓解延迟
- **优点**：工程成本低，延迟可控
- **缺点**：pivot 和重复问题无法根本解决

---

## 7. 建议

基于以上分析，建议 v0.0.4 不直接切换翻译模型，而是：

1. **保留当前 opus-mt + 已实现的四项优化**（动态长度、重复截断、warmup、缓存）
2. **v0.0.5 启动 m2m-100 418M 的 fine-tuning 实验**，目标不是部署，而是：
   - 验证在影视字幕数据上的质量提升
   - 评估 fine-tuned 模型的本地延迟
3. **同步准备蒸馏方案**：若 m2m-100 延迟过高，将其蒸馏为 100M–200M 学生模型
4. **在有稳定 Hugging Face 访问的环境中补充实测**
